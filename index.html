Planning for post launch

  • Make it to launch

    • Move non-critical bugs and feature requests to the backlog
    • Viable launch candidate
  • Discuss support early in the development process

Make it to launch

  • Test the production environments early in the project
  • Version control and backup services
  • Standardize launch requirements and deployment procedures

Plan for Phase 2: After the launch

  • Review the backlog
  • Establish a deployment process
  • Managed hosting vs. Unmanaged Hosting

Discuss support early in the development process

  • Establish requirements before beginning requirements
  • Discovery time is a real thing and is highly encouraged
  • Set expectations early and keep an open line of communication with the stake holders

Design for maintainability

Do your research before committing a module to a project

  • Project Activity
  • Module Compatibility
  • Resource control and module overhead

Project Activity

  • How many sites use the module?
  • How many downloads?
  • Issue queue activity

Module Compatibility

Common contributed modules

  • Field UI
  • Views
  • Features + Strongarm
  • Rules
  • Feeds

Resource control and module overhead

  • Don't get too carried away with "magic" modules
  • Monitor performance throughout the project
    • New Relic - Great for monitoring performance
  • Use the Features module to export functionality from the database into code
  • Cache all the things!
    • Cloudflare - CDN, Security, Analytics, and more!

Contributed modules vs. custom modules

  • Both have benefits
  • Who maintains the code?
  • Developer familiarity

Contributed modules

  • Community support
  • Module familiarity among developers
  • Typically more cost effective

Custom modules

  • High-granularity with data and site content
  • Easier to control resources
  • Who will responsible for maintaining the code?
  • Often is more costly and time-intensive than using a contributed module

Dealing with security updates

  • Maintain an healthy codebase
  • Document patches and lock future updates
  • Establish a release plan

Maintain an healthy codebase

  • Don't hack modules
    • Leverage the Drupal API
  • Onboarding external sites
    • Hacked / Coder - Audit site code for overrides and coding standards.
  • Keep modules in relevant directories
      • Example:
    • sites/all/modules/contrib
    • sites/all/modules/features
    • sites/all/modules/custom
    • sites/all/modules/patched

Document patches and lock future updates

  • Locking updates with drush
    • Prevents drush from automatically updating modules with drush up
    • Review previous locks. Previous issues may have been resolved
  • Keep up to date update documentation for a project

Establish a release plan

  • Do you have access to the hosting environment?
  • Who is delivering the code?
  • How would you deal with deployment mishaps?

Establish a standardized workflow

  • Who is the release manager?
  • Establish a git flow that makes sense
  • Working with third parties

Quality assurance and automated testing

  • Automated Testing
    • Behat
    • Selenium
  • Manual Quality Assurance
  • Limit client interaction between different phases of quality assurance testing

Development Environments

  • Shared development environments
  • Local environments
  • Centralized integration points to test deployments prior to staging

Deploying bug fixes, feature requests and code updates

  • Information gathering and documentation
  • Can the issue be reproduced?
  • Version control is your friend, be comfortable and git familiar

Information gathering and documentation

  • Debug Tools - Easy reporting for the client, and effective information gathering
  • Establish acceptance criteria and budgets prior to starting new tasks
  • Testing instructions that match acceptance criteria

Can the issue be reproduced?

  • Reproduce on production and development environments
  • Review recent updates
  • Once resolved, document the fix and make the team aware

Version control is your friend

  • Version control is free, you should be using it.
  • Drupal Modules
    • Features/Strongarm (Drupal 7)
    • Configuration Management (Drupal 8)
  • Branches, Pull Requests, and Rebasing

Knowing your team and your limitations

  • Know your team online and offline
  • Let developers specialize in the preferred area of focus
  • Respect a developers time and life

Getting to know your team

  • Drupal Camps, Drupalcon, and meetups are great for teams and meeting new friends
  • How did they get to where they are today?
  • What are they into outside of Drupal?

Let developers specialize

  • Use developer time more efficiently
  • Don't overload developers
  • Happy developers leads to happy clients

Respect a developer's time

  • Avoid overloading a developer
  • Standardization helps decrease ramp up time switching between projects
  • Open line of communication, set expectations early
